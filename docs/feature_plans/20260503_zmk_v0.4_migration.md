## Background

ZMK `main` moved to Zephyr 4.1 in December 2025, introducing Hardware Model v2 (HWMv2) â€” a fundamental restructuring of how boards are defined. All out-of-tree board definitions must be migrated. Shields do **not** require HWMv2 changes but have other breaking changes (Cirque driver, Kconfig removals, etc.).

This migration spans **four repositories**:
- `zmk-fingerpunch-keyboards` (this repo)
- `zmk-fingerpunch-controllers`
- `zmk-fingerpunch-vik`
- `zmk-config-fp-example` (user-facing example/template repo)

**Note:** Hundreds of users have forked or followed `zmk-config-fp-example` to create their own `zmk-config` repos. Every breaking change in the upstream repos has a downstream ripple effect on those users. This plan must account for user-facing communication and migration guidance.

**Reference:** https://zmk.dev/blog/2025/12/09/zephyr-4-1

## Breaking Changes Requiring Action

### 1. HWMv2 Board Migration (ALL boards across keyboards + controllers repos)

**What changed:** Boards no longer live under `boards/arm/`. They must be placed in a vendor-named directory (e.g., `boards/fingerpunch/`) and require a `board.yml` manifest file.

**Required for each board:**
- Move from `boards/arm/<board>/` â†’ `boards/fingerpunch/<board>/`
- Create `board.yml` with board name, vendor, SoC, revisions, and optional variants
- Rename `Kconfig.board` â†’ `Kconfig.<board>`
- In `Kconfig.<board>`: Remove `bool` line, change `depends on` â†’ `select`
- In `<board>_defconfig`: Remove `CONFIG_SOC_SERIES_*`, `CONFIG_SOC_NRF*`/`CONFIG_SOC_RP2040`, `CONFIG_BOARD_*`
- Add ZMK board variant support (add `zmk` variant to `board.yml` socs)

**Note:** The Zephyr `board_v1_to_v2.py` script does NOT work well with split boards â€” **ximi must be migrated manually**.

### 2. nRF52840 DC/DC Mode (ffkb_holyiot_v1, vikoto)

**What changed:** DC/DC mode is now configured in devicetree, not Kconfig.

- Remove `BOARD_ENABLE_DCDC` / `SOC_DCDC_NRF52X` from `Kconfig` and `Kconfig.defconfig`
- Add `&reg1 { regulator-initial-mode = <NRF5X_REG_MODE_DCDC>; };` to board `.dts`

### 3. nRF52840 NFC Pins as GPIO

**What changed:** `CONFIG_NFCT_PINS_AS_GPIOS` Kconfig removed; use devicetree instead.

- If any board/shield uses NFC pins as GPIO, add `&uicr { nfct-pins-as-gpios; };` to devicetree
- Remove any `CONFIG_NFCT_PINS_AS_GPIOS=y` from Kconfig/defconfig (none found currently, but verify)

### 4. RP2040 Devicetree Changes (pinkies_out_v3, vulpes_majora_v1, ximi, svlinky, xivik)

**What changed:** RP2040 include path changed, fixed-clock node removed, clock control required.

- Change `#include <rpi_pico/rp2040.dtsi>` â†’ `#include <raspberrypi/rpi_pico/rp2040.dtsi>`
- Remove `xtal-clk` / `fixed-clock` node from DTS
- Add `CONFIG_CLOCK_CONTROL=y` to defconfig
- Add `&timer { status = "okay"; };` and `&rtc` / `&vreg` nodes to DTS
- Add `zephyr,flash-controller = &ssi;` to chosen node

### 5. Cirque Pinnacle Driver Migration (zmk-fingerpunch-vik + shields in keyboards)

**What changed:** Upstream Zephyr now includes the Pinnacle driver. The `petejohanson/cirque-input-module` module must be removed.

**Property renames:**
| Old (out-of-tree) | New (upstream Zephyr) |
|---|---|
| `dr-gpios` | `data-ready-gpios` |
| `no-taps` | *(remove; use `primary-tap-enable` to opt in)* |
| `x-invert` / `y-invert` | `invert-x` / `invert-y` |
| `rotate-90` | `swap-xy` |
| `sleep` | `sleep-mode-enable` |

**Files affected:**
- `zmk-fingerpunch-keyboards`: `ffkb_v2.overlay`, `ffkb_we_v1.overlay`, `pinkies_out_v2.overlay`
- `zmk-fingerpunch-vik`: `vik_cirque_i2c.overlay`, `vik_cirque_spi.overlay`, `vik_cirque_spi_split_peripheral.overlay`
- `zmk-config-fp-example`: `config/ffkb_v2.overlay` (user-facing example with `dr-gpios`, `sleep`, `no-taps`, and `sensitivity` properties)

**Note on `zmk,input-listener` properties:** The properties `xy-swap`, `y-invert`, `x-invert` used on `zmk,input-listener` nodes (e.g., in `zmk-config-fp-example`) are ZMK-specific, NOT Cirque driver properties. These should be **unchanged** in ZMK v0.4 but verify.

**Note on `compatible` string:** Verify whether `compatible = "cirque,pinnacle"` remains valid for both I2C and SPI, or if it needs to change to `cirque,pinnacle-i2c` / `cirque,pinnacle-spi`.

**Note on `sensitivity` property:** The `zmk-config-fp-example` uses `sensitivity = "4x"` â€” verify this property exists in the upstream Zephyr Pinnacle driver.

**Module removal:**
- Remove `cirque-input-module` from `zmk-fingerpunch-vik/config/deps.yml`
- Users who manually added `cirque-input-module` to their own `west.yml` (following old zmk-config-fp-example README) must also remove it

### 6. CONFIG_WS2812_STRIP Removal

**What changed:** Kconfig flag removed; SPI is auto-enabled.

### 13. Bootloader / Boot Retention Setup (ALL boards)

**What changed:** The previous method to enable `&bootloader` has been disabled. ZMK now requires "boot retention" to be configured for the bootloader reset behavior to work.

- All custom boards need boot retention set up per: https://zmk.dev/docs/development/hardware-integration/bootloader
- Without this, `&bootloader` behavior in keymaps will stop working
- Affects: `ffkb_holyiot_v1`, `vikoto`, `svlinky`, `xivik`, `pinkies_out_v3`, `vulpes_majora_v1`, `ximi`

### 15. Vikoto Charger Current Kconfig Preservation

**What changed:** The vikoto `Kconfig` file contains BOTH DC/DC settings (to be removed) AND charger current settings (`BOARD_VIKOTO_CHARGER_CURRENT_*`) that **must be preserved**. The charger current options are used by `pinmux.c` to configure GPIO pins for battery charging.

- Remove ONLY the `BOARD_ENABLE_DCDC` / `SOC_DCDC_NRF52X` sections from `Kconfig`
- **Keep** the `BOARD_VIKOTO_CHARGER_CURRENT` choice and all its options
- Verify `pinmux.c` still compiles â€” it uses `#if CONFIG_BOARD_VIKOTO` and `CONFIG_BOARD_VIKOTO_CHARGER_CURRENT_*` macros
- In HWMv2, `CONFIG_BOARD_VIKOTO` is still auto-set by the build system, so `pinmux.c` should work without changes
- The `CMakeLists.txt` (`zephyr_library_sources(pinmux.c)`) should also remain functional

## Key Findings & Lessons Learned (from implementation)

### Boot Retention Requires Kconfig `imply` Lines
Including the boot retention DTS file (e.g., `nrf52840_uf2_boot_mode.dtsi` or `rp2040-boot-mode-retention.dtsi`) is **not sufficient**. The board's `Kconfig.<board>` must also contain:
```
imply RETAINED_MEM
imply RETENTION
imply RETENTION_BOOT_MODE
```
Without these, the linker fails with undefined references to `retention_read`, `retention_write`, `retention_clear`. This was discovered from the `puchi_ble` reference board in ZMK.

### `board.yml` Schema: `revision:` vs `revisions:`
The board-level key is `revision:` (singular), which contains a nested `revisions:` array. Using `revisions:` (plural) at the top level causes a schema validation error.

### Split Boards with `boards:` (plural) Need Separate Kconfig Files
When using `boards:` (plural) in `board.yml` (for split boards like ximi), each board entry needs its own `Kconfig.<board_name>` file (e.g., `Kconfig.ximi_left` and `Kconfig.ximi_right`). A single `Kconfig.ximi` does NOT work.

### `CONFIG_WS2812_STRIP` â†’ `CONFIG_WS2812_STRIP_SPI`
The old symbol is **undefined** in Zephyr 4.1 (not just deprecated). Kconfig aborts with "undefined symbol" error. Must be changed to `CONFIG_WS2812_STRIP_SPI=y`.

### `CONFIG_BT_CTLR` Deprecated
Must be removed from `Kconfig.defconfig` files. Causes deprecation warning but not build failure.

### KSCAN Deprecation Warning
The KSCAN deprecation warning is a ZMK-level issue (ZMK itself `select KSCAN`). Not fixable at board level â€” can be safely ignored.

### `seeeduino_xiao` Board Name Unchanged
While `seeeduino_xiao_ble` â†’ `xiao_ble` and `seeeduino_xiao_rp2040` â†’ `xiao_rp2040`, the original SAMD21 `seeeduino_xiao` board name is **unchanged** in ZMK v0.4.

### `nice_nano_v2` â†’ `nice_nano`
v2 is now the default revision. The `//zmk` suffix is NOT needed for ZMK in-tree boards. Several shields had both `nice_nano.overlay` (v1) and `nice_nano_v2.overlay` â€” v1 files were typically stubs or identical to v2 and were deleted.

### Cirque `compatible` String Unchanged
`compatible = "cirque,pinnacle"` works for both I2C and SPI. No need to change to `cirque,pinnacle-i2c` or `cirque,pinnacle-spi`.

### `cirque-input-module` No Longer Needed
The Pinnacle driver is now in the Zephyr tree. The external `cirque-input-module` dependency can be removed from `deps.yml`.

### RP2040 `CONFIG_CLOCK_CONTROL=y` Required
Must be added to all RP2040 board defconfigs. Without it, clock initialization fails.
